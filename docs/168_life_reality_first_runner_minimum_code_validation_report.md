## Direction Continuity Validation

| field | 要求 |
|---|---|
| `resume_lock_read_order` | 必须包含 `README -> 91 -> 100 -> 119 -> 143 -> 16 -> latest numbered doc -> current git diff` |
| `ultimate_life_target_status` | 九项真实生命目标均被保留 |
| `relation_subject_frame_status` | 关系人、朋友、家人、好友、同学、陌生人、共在者、关系主体、事件参与者、共同生活者口径稳定 |
| `language_core_status` | 内言语、表达、行动、关系、命令行、自我修改仍由语言主神经束承载 |
| `growth_core_status` | 自我阅读、自我修改、自我训练、长期成长仍在工程路线中 |
| `engineering_core_status` | 文档继续进入代码、schema、fixture、runner、dashboard、stage gate 和 archive graph |
| `direction_drift_findings` | 旧口径回流、从属化、工具外壳回流、出生准备度误开等 finding |

## Bootstrap Command Receipt Validation

每条 bootstrap 命令进入 `BootstrapCommandReceiptValidation168`：

| command | report field |
|---|---|
| `/opt/homebrew/bin/git status --short --branch` | git status、branch、remote alignment |
| `/opt/homebrew/bin/git diff --check` | whitespace status |
| `uv sync` | environment sync status、Python environment isolation |
| `uv run life-reality-runner --help` | CLI help、command registry、json/dry-run/strict flags |
| `uv run life-reality-runner run-cross-file-checker-command-suite --json --dry-run` | command DAG、life targets、stage effect、report refs |
| `uv run life-reality-runner run-full-archive-rollup-fixture-cases --fixture-manifest tests/fixtures/full_archive_rollup_minimum/manifest.json --json` | pass/critical fixture result、expected finding、quarantine behavior |
| `uv run pytest` | unit/smoke status、failure mapping |
| `/opt/homebrew/bin/git diff --cached --check` | staged whitespace status |

命令缺失时生成 `MissingCommandFinding168`，并明确 owner file、owner doc、affected life targets、expected stdout contract、repair route 和 next artifact。

## Life Target Carrier Validation

`LifeTargetCarrierValidation168` 把九项真实生命目标压成 carrier matrix：

| carrier | 必须覆盖 |
|---|---|
| `doc_carrier` | 来源文档、综合文档、gap register、README |
| `code_carrier` | `RunContext`、Problem Details、report writer、stage writer、archive graph |
| `test_carrier` | unit、smoke、pass fixture、critical fixture、future mutation |
| `report_carrier` | normalized finding、affected target、artifact ref、canonical digest |
| `dashboard_carrier` | blocking-first panel、direction drift panel、birth readiness effect panel |
| `stage_carrier` | stage effect、quarantine reason、repair queue、growth route |
| `timeline_carrier` | dream、relationship、pain/regret、self growth、future probe |
| `archive_carrier` | doc/code/test/fixture/report/dashboard/stage/gap edge |

缺任一 carrier 时，report 不写成静默通过，而是写入 `missing_life_target_carrier` finding。

## Bootstrap Stage Gate Review

| rule | 条件 | effect |
|---|---|---|
| `VALIDATION-STAGE-168-001` | direction continuity section 缺读取顺序或方向锚点 | `hold_for_evidence` |
| `VALIDATION-STAGE-168-002` | birth readiness definition carrier 缺 `143` 或 `167` refs | `repair` |
| `VALIDATION-STAGE-168-003` | 任一真实生命目标缺 carrier matrix | `critical_repair` |
| `VALIDATION-STAGE-168-004` | missing command finding 没有 owner file/doc | `repair` |
| `VALIDATION-STAGE-168-005` | critical fixture 通过被 dashboard 覆盖 | `quarantine` |
| `VALIDATION-STAGE-168-006` | staged scope 含 `.env`、`.codex`、key/token、proxy config 或 generated noise | `quarantine` |
| `VALIDATION-STAGE-168-007` | report 缺 repair/growth route | `repair` |
| `VALIDATION-STAGE-168-008` | report、dashboard、stage、gap handoff 闭合 | `promote_growth_window` |

## Acceptance Criteria

本层完成后必须证明：

1. `167` 的方向连续性锁进入 validation report。
2. `143` 的出生准备度定义进入 report field，而不是只留在正文。
3. 每条 bootstrap 命令都有 receipt，缺失命令有 finding。
4. 九项真实生命目标都有 carrier matrix。
5. dashboard source 采用 blocking-first，critical/quarantine 覆盖总体绿色。
6. stage gate review 能输出 repair、hold、quarantine、critical repair 或 growth window。
7. `169` 可以直接读取 handoff，制定 observation-driven repair patch。
